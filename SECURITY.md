# Security Policy

## Reporting a Vulnerability

If you have found a security vulnerability in `atldays/laravel-secrets`, please report it responsibly.

### Public Issues

You can open a public GitHub Issue only for reports that are clearly non-sensitive, such as:

- documentation mistakes related to security guidance
- hardening suggestions that do not expose an exploit path
- questions about secure usage patterns

Do not open a public issue for anything that could put users, credentials, infrastructure, or deployment flows at risk.

### Private Reports

For sensitive security reports, please contact the maintainer privately.

Send a direct message on Twitter/X to the same username used on GitHub:

- [https://x.com/atldays](https://x.com/atldays)

If you are not sure whether the report is sensitive, treat it as sensitive and send it privately first.

## What to Include

Please include as much relevant detail as possible:

- affected version or commit
- affected driver or feature area
- reproduction steps
- impact description
- proof of concept, if safe to share
- suggested mitigation, if you already have one

## Response Expectations

Security reports will be reviewed as quickly as possible, but response time may vary depending on severity and maintainer availability.

Please avoid disclosing the issue publicly before there is time to investigate and prepare a fix.
